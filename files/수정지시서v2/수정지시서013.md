# 수정지시서 013: POST 핸들러에서 request body 필드를 c.Query()로 읽는 버그

## 현상

POST 핸들러에서 `request.BidAmount`처럼 request body에서 와야 할 필드를 `c.Query("BidAmount")`(query parameter)로 읽고 있음.

### 정상 케이스 (CreateGig)

SSaC 스펙:
```go
// @post Gig gig = Gig.Create({Budget: request.Budget, Title: request.Title, Description: request.Description})
```

생성된 코드 (정상):
```go
var req struct {
    Budget      int64  `json:"Budget"`
    Title       string `json:"Title"`
    Description string `json:"Description"`
}
if err := c.ShouldBindJSON(&req); err != nil {
    c.JSON(http.StatusBadRequest, gin.H{"error": "invalid request body"})
    return
}
budget := req.Budget
```

### 문제 케이스 (SubmitProposal)

SSaC 스펙:
```go
// @get Gig gig = Gig.FindByID({ID: request.ID})
// @empty gig "Gig not found"
// @auth "submit_proposal" "gig" {id: gig.ID} "Not authorized"
// @post Proposal proposal = Proposal.Create({GigID: gig.ID, FreelancerID: currentUser.ID, BidAmount: request.BidAmount})
```

생성된 코드 (버그):
```go
bidAmount, err := strconv.ParseInt(c.Query("BidAmount"), 10, 64)
if err != nil {
    c.JSON(http.StatusBadRequest, gin.H{"error": "BidAmount: 유효하지 않은 값"})
    return
}
```

## 원인 분석

`@get`에서 `request.ID`를 path param으로 이미 처리한 후, `@post`에서 `request.BidAmount`를 만나면 body 바인딩 없이 `c.Query()`로 fallback하는 것으로 추정.

`CreateGig`처럼 `request.*` 필드가 모두 `@post`에서만 참조되면 `ShouldBindJSON`으로 올바르게 생성됨.

## 수정 방향

POST/PUT 핸들러에서 `request.*` 필드 중 path param이 아닌 것은 반드시 JSON body에서 읽어야 함:
1. `@get`에서 `request.ID` → `c.Param("ID")` (path param) — 현재 정상
2. `@post`에서 `request.BidAmount` → `c.ShouldBindJSON(&req)` 후 `req.BidAmount` — 현재 버그

path param으로 이미 처리된 필드를 제외한 나머지 `request.*` 필드는 `ShouldBindJSON` 구조체에 포함시켜야 함.

## 검증 방법

```bash
# SubmitProposal 생성 코드에서 c.Query 대신 ShouldBindJSON이 사용되는지 확인
ssac gen specs/dummy-gigbridge/service/proposal/submit_proposal.go
```

생성된 코드에 `c.ShouldBindJSON(&req)` + `req.BidAmount` 패턴이 나와야 함.
