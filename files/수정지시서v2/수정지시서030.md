# 수정지시서 030: buildArgsCodeFromInputs 알파벳순 정렬 → 모델 시그니처 순서

## 문제

`buildArgsCodeFromInputs`가 Inputs map의 key를 알파벳순으로 정렬하여 모델 호출 인자를 생성한다. 그러나 모델 시그니처(ssac-model gen)는 SQL 파라미터 순서($1, $2, $3)대로 생성된다. 두 순서가 다르면 인자가 뒤바뀐다.

### 재현

DDL: `INSERT INTO users (email, password_hash, name)` → 시그니처 `Create(email, passwordHash, name)`

SSaC: `@post User user = User.Create({Email: request.email, PasswordHash: hp.HashedPassword, Name: request.name})`

현재 생성: `h.UserModel.Create(email, hp.HashedPassword, name)` — 알파벳순 (E < N < P)

기대: `h.UserModel.Create(email, hp.HashedPassword, name)` — 이 경우 우연히 맞지만, `{Name: ..., Email: ..., PasswordHash: ...}` 같이 필드 순서가 다르면 항상 알파벳순으로 나와서 SQL 순서와 불일치.

### 영향

- 타입이 모두 `string`이면 컴파일 통과, 런타임 데이터 오염
- `int64`와 `string`이 섞이면 컴파일 에러

## 근본 원인

SSaC Inputs는 `map[string]string`이라 순서가 없다. `buildArgsCodeFromInputs`가 결정론적 출력을 위해 알파벳순 정렬을 사용하지만, 모델 시그니처는 SQL 파라미터 순서를 따른다.

## 수정 방안

### 방안 A: 파서에서 순서 보존

파서가 SSaC `{Key1: val1, Key2: val2}` 를 파싱할 때 선언 순서를 보존한다. `Inputs`를 `map[string]string`에서 `[]InputPair` (ordered slice)로 변경하거나, 별도 `InputOrder []string`을 추가한다. SSOT 작성자가 모델 시그니처 순서에 맞춰 작성하면 된다.

### 방안 B: 코드젠에서 모델 시그니처 참조

`buildArgsCodeFromInputs`가 모델 시그니처의 파라미터 순서를 참조하여 인자를 배치한다. SSaC Inputs의 key가 모델 파라미터명과 매칭되므로, 시그니처 순서대로 정렬 가능. 단, 코드젠 시점에 모델 시그니처 정보가 필요.

### 방안 C: Inputs key를 모델 파라미터명으로 매칭 (현실적)

`buildArgsCodeFromInputs`에 모델 파라미터 순서 정보를 전달한다. Inputs map의 key(= 모델 파라미터명)를 시그니처 순서대로 배치.

## 수정지시서 029와의 관계

029에서 `query` 값을 마지막으로 보내는 특수 처리를 했는데, 이 수정이 완료되면 029의 query 특수 처리도 자연스럽게 통합된다 — query 파라미터도 모델 시그니처에서 마지막이므로.

## 테스트 방법

1. DDL `(email, password_hash, name)` 순서의 테이블 + SSaC @post에서 알파벳순과 다른 순서로 Inputs 작성
2. 생성된 코드가 모델 시그니처 순서와 일치하는지 확인
3. 기존 테스트 전체 통과 확인
