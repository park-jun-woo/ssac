# 수정지시서 020: @subscribe 메시지 타입 + 코드젠 분리

## 배경

수정지시서 019에서 `@subscribe` 트리거와 `@publish` 시퀀스를 추가했다. 현재 문제:

1. `@subscribe` 함수도 `func Name(c *gin.Context)` gin handler로 생성됨 — 비동기 큐 핸들러에 HTTP context가 부자연스러움
2. `message` 변수가 선언 없이 `message.Field` 형태로 출력됨 — 컴파일 불가
3. 메시지 페이로드 구조가 토픽마다 다른데 타입 정보가 없음

## 변경 요청

### 1. 메시지 타입을 Go struct로 선언

`.ssac` 파일에 Go struct를 직접 작성한다. SSaC DSL이 아닌 표준 Go 문법.

```go
package service

type OnOrderCompletedMessage struct {
    OrderID int64
    Email   string
    Amount  int64
}

// @subscribe "order.completed" OnOrderCompletedMessage
// @get Order order = Order.FindByID({ID: message.OrderID})
// @call mail.SendEmail({To: message.Email, Subject: "주문 완료"})
// @put Order.UpdateNotified({ID: order.ID, Notified: "true"})
func OnOrderCompleted(OnOrderCompletedMessage message) {}
```

이점:
- Go AST로 파싱 — 새 DSL 파서 불필요
- 복잡한 타입 대응 가능 (중첩 struct, slice 등)
- struct를 별도 파일로 분리해도 검증 가능
- IDE 자동완성, 타입체크 그대로 동작

### 2. @subscribe 문법 변경

기존:
```
@subscribe "topic"
```

변경:
```
@subscribe "topic" TypeName
```

- `TypeName` — 메시지 struct 타입명 (같은 패키지 또는 별도 파일에 선언)

### 3. 함수 시그니처 — 파라미터에 메시지 타입 명시

```go
func OnOrderCompleted(OnOrderCompletedMessage message) {}
```

- 파라미터: `TypeName variableName` 형태
- 변수명은 반드시 `message` (예약 변수)
- 파서가 파라미터에서 타입과 변수명을 추출하여 검증에 사용

### 4. @subscribe 함수 코드젠 분리

HTTP 함수:
```go
func CompleteOrder(c *gin.Context) {
    // request 바인딩, c.JSON 응답
}
```

Subscribe 함수:
```go
func OnOrderCompleted(ctx context.Context, message OnOrderCompletedMessage) error {
    // message.OrderID 직접 접근
    // 에러 시 return err
    // 성공 시 return nil
}
```

차이점:

| 항목 | HTTP 함수 | Subscribe 함수 |
|---|---|---|
| 시그니처 | `func(c *gin.Context)` | `func(ctx context.Context, message T) error` |
| 입력 | `request` (gin binding) | `message` (함수 파라미터) |
| 출력 | `c.JSON()` | `return error` |
| 에러 처리 | `c.JSON(500, ...)` + `return` | `return fmt.Errorf(...)` |

### 5. 시퀀스 코드젠 변경 (subscribe 함수 내부)

subscribe 함수 내부의 시퀀스는 gin context 대신 `ctx`를 사용한다.

| 시퀀스 | HTTP 함수 | Subscribe 함수 |
|---|---|---|
| `@get` | `courseModel.FindByID(...)` | 동일 |
| `@put` | `courseModel.Update(...)` | 동일 |
| `@call` | `mail.SendEmail(...)` | 동일 |
| `@publish` | `queue.Publish(c.Request.Context(), ...)` | `queue.Publish(ctx, ...)` |
| `@empty` | `c.JSON(404, ...)` + `return` | `return fmt.Errorf(...)` |
| 에러 처리 | `c.JSON(500, ...)` + `return` | `return fmt.Errorf(...)` |

### 6. 파서 출력 변경

`SubscribeInfo`에 타입명 추가:

```go
type SubscribeInfo struct {
    Topic       string // "order.completed"
    MessageType string // "OnOrderCompletedMessage"
}
```

`ServiceFunc` 파라미터 정보:

```go
type ServiceFunc struct {
    Name      string
    Subscribe *SubscribeInfo
    Param     *ParamInfo       // nil이면 HTTP (파라미터 없음)
    // ...
}

type ParamInfo struct {
    TypeName string // "OnOrderCompletedMessage"
    VarName  string // "message"
}
```

### 7. 검증 규칙

| 규칙 | Level |
|---|---|
| `@subscribe` 함수에 파라미터 없음 | ERROR |
| 파라미터 변수명이 `message`가 아님 | ERROR |
| 파라미터 타입이 패키지 내 struct로 존재하지 않음 | ERROR |
| `message.X`의 X가 struct 필드에 없음 | ERROR |
| struct 코드젠 출력에서 `json` 태그 누락 | — (코드젠이 자동 추가) |


테스트 케이스:
- `@subscribe "topic" TypeName` 파싱 — topic + MessageType 추출
- `func Name(TypeName message) {}` 파라미터 파싱 — TypeName + VarName 추출
- 메시지 struct → Go AST 파싱으로 필드 정보 수집
- `message.X` 필드가 struct에 존재하는지 검증
- subscribe 함수 시그니처 코드젠 (`ctx context.Context, message T`) error`
- subscribe 함수 내 에러 처리 (`return err` vs `c.JSON`)
- `@publish` 코드젠 — subscribe 함수 내에서 `ctx` 사용
- 파라미터 누락 → ERROR
- 파라미터 변수명 `message` 아님 → ERROR
