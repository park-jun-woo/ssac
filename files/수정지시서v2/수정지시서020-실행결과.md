### 4. generator/go_target.go

**GenerateFunc() 분기**:

```go
func (g *GoTarget) GenerateFunc(sf parser.ServiceFunc, st *validator.SymbolTable) ([]byte, error) {
    if sf.Subscribe != nil {
        return g.generateSubscribeFunc(sf, st)
    }
    return g.generateHTTPFunc(sf, st)
}
```

기존 로직을 `generateHTTPFunc()`으로 이동.

**`generateSubscribeFunc()`**: subscribe 함수 전용 코드젠:

- 시그니처: `func Name(ctx context.Context, message T) error`
- import: `context`, `fmt` + 시퀀스별 필요 패키지
- 에러: `return fmt.Errorf("message: %w", err)`
- 성공: `return nil`
- gin 의존성 없음

**`subscribeTemplateName()`**: `sub_` 접두사 템플릿 이름 반환.

**`collectSubscribeImports()`**: subscribe 함수 전용 import 수집.

## 주요 설계 결정

1. **Go 문법 준수**: 수정지시서의 `func Name(TypeName message) {}`는 Go 문법상 `func Name(message TypeName) {}`이 올바름. Go AST 파싱을 사용하므로 Go 문법을 따름.
2. **Target 필드 재활용**: `@subscribe` 파싱 시 `Sequence.Target`에 MessageType을 임시 저장. subscribe는 시퀀스에서 제거되므로 충돌 없음.
3. **HTTP 코드 무영향**: `GenerateFunc()` 분기로 기존 HTTP 코드젠은 `generateHTTPFunc()`으로 그대로 유지. 기존 31개 generator 테스트 모두 통과.
4. **struct 범위**: 같은 .ssac 파일 내 struct만 수집. 파일 레벨이므로 같은 파일의 모든 함수가 동일한 Structs 공유.
5. **sub_ 템플릿**: HTTP와 subscribe 에러 처리가 근본적으로 다르므로 (c.JSON vs return err) 별도 템플릿으로 분리. 모델 호출 로직은 동일하나 에러 핸들링만 다름.
