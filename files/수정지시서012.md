# SSaC codegen 수정 요구서 — @sequence call 코드 생성

**요청자**: fullend
**일자**: 2026-03-09
**우선순위**: 긴급 (생성된 코드가 컴파일 안 됨)

## 배경

fullend Phase 26에서 `pkg/` func 파일들의 `*Input/*Output` → `*Request/*Response` 명명 변경을 완료했다.
이후 dummy-lesson 프로젝트로 `fullend gen` → 서버 빌드 → Hurl 스모크 테스트를 실행한 결과,
SSaC codegen이 생성하는 `@sequence call` 코드에서 **5건의 컴파일 에러**가 발생했다.

## 수정 항목

### 1. import 경로: spec 파일의 import 선언을 사용

**현재 동작** (`go_target.go:464`):
```go
case seq.Type == parser.SeqCall && seq.Package != "":
    seen[seq.Package] = true  // "auth" → import "auth"
```

**생성 결과**: `import "auth"` → Go 1.25에서 stdlib `auth` 패키지와 충돌

**원인**: SSaC codegen이 `@func auth.verifyPassword`에서 `auth`를 패키지 경로로 직접 사용한다. 하지만 `auth`는 패키지 alias일 뿐, 실제 import 경로는 spec 파일의 Go import 블록에 사용자가 직접 선언한다.

**기대 동작**: spec 파일에 사용자가 import를 명시적으로 작성하고, codegen은 그 import를 그대로 사용한다.

```go
// spec 파일
package service

import (
    "net/http"
    "github.com/geul-org/fullend/pkg/auth"
)

// @sequence call
// @func auth.verifyPassword
// ...
func Login(w http.ResponseWriter, r *http.Request) {}
```

**해결 방안**: `collectImports()`에서 `@func`의 패키지명으로 import를 자동 추가하지 말고, spec 파일의 Go import 선언을 파싱하여 그대로 전달한다. `@func auth.xxx`의 `auth`는 import된 패키지의 alias로 간주하고, 코드 생성 시 해당 alias를 사용하면 된다.

SSaC parser가 spec 파일의 import 블록을 이미 Go AST로 파싱하고 있으므로, 그 import 목록을 generator에 전달하여 출력 파일의 import에 포함시키면 된다.

### 2. struct 이름: `*Input` → `*Request`

**현재 동작** (`go_templates.go:90`):
```
{{.PkgName}}.{{.FuncMethod}}Input{ {{.InputFields}} }
```

**생성 결과**: `auth.HashPasswordInput{...}` — 존재하지 않는 타입

**기대 동작**: `auth.HashPasswordRequest{...}`

**해결 방안**: 템플릿에서 `Input` → `Request` 변경
```
{{.PkgName}}.{{.FuncMethod}}Request{ {{.InputFields}} }
```

### 3. 결과 없는 call의 반환값 처리

**현재 동작**: `@sequence call`에 `@result`가 없으면 `err :=` 또는 `err =` 생성

**생성 결과**:
```go
err = auth.VerifyPassword(auth.VerifyPasswordInput{...})
```

**실제 함수 시그니처**: `func VerifyPassword(req VerifyPasswordRequest) (VerifyPasswordResponse, error)`
→ 항상 `(Response, error)` 반환

**기대 동작**:
```go
_, err = auth.VerifyPassword(auth.VerifyPasswordRequest{...})
```

**해결 방안**: `@result`가 없어도 func 호출은 항상 `(Response, error)` 반환이므로 `_, err` 패턴 사용

### 4. 결과 추출 필드명: `ucFirst(var)` → Response struct 필드명

**현재 동작** (`go_target.go:264-268`):
```go
d.ResultField = ucFirst(seq.Result.Var)
// @result token Token → ResultField = "Token"
```

**생성 결과**:
```go
out, err := auth.IssueToken(auth.IssueTokenInput{...})
token := out.Token  // Token 필드 없음
```

**실제 IssueTokenResponse**: `AccessToken string` 필드를 가짐

**기대 동작**:
```go
out, err := auth.IssueToken(auth.IssueTokenRequest{...})
token := out.AccessToken
```

**해결 방안**: `@result` 구문에서 추출할 필드명을 명시적으로 지정할 수 있어야 한다.

방안 A — `@result` 뒤에 필드명 지정:
```go
// @result token Token.AccessToken
// → token := out.AccessToken
```

방안 B — Response에 필드가 1개면 자동 추출, 여러 개면 명시 필요

방안 C — 기존 방식 유지하되, funcspec의 Response struct 정의에서 필드명을 읽어서 매칭

**추천**: 방안 A가 가장 명시적이고 간단하다.

### 5. Input struct 필드명 매칭

**현재 동작** (`go_target.go:537-560`):
```go
// @param user.ID → fieldName="ID", fieldValue="user.ID"
// → {ID: user.ID}
```

**실제 IssueTokenRequest**: `UserID int64` 필드를 가짐 (not `ID`)

**생성 결과**: `auth.IssueTokenInput{ID: user.ID}` — 컴파일 에러

**기대 동작**: `auth.IssueTokenRequest{UserID: user.ID}`

**해결 방안**: `@param`에서 Request struct 필드명을 명시적으로 지정할 수 있어야 한다.

현재 SSaC는 `@param` dot notation에서 dot 뒤의 부분을 struct 필드명으로 그대로 사용한다.
하지만 Request struct의 필드명과 소스 변수의 필드명이 다를 수 있다.

방안 A — `->` 매핑 (DDL column 매핑과 동일 문법):
```go
// @param user.ID -> UserID
// → {UserID: user.ID}
```

방안 B — positional matching (순서로 매칭, 이름 무시)

**추천**: 방안 A — 기존 `->` 문법과 일관성 있고 명시적이다.
